# 09 测试计划与验收标准

## 1. 测试目标

验证系统是否满足老师要求：

- 能管理学员、教练、车辆；
- 能进行练车预约；
- 能控制教练每日容量；
- 能处理主教练满员后的调剂；
- 能避免重复预约和超额预约；
- 能完成练车记录；
- 能体现信息化和资源均衡价值。

## 2. 测试范围

| 类型 | 内容 |
|---|---|
| 功能测试 | 登录、管理、排班、预约、取消、调剂、完成 |
| 权限测试 | 学员、教练、管理员权限隔离 |
| 业务规则测试 | 容量限制、重复预约、车辆状态 |
| 并发测试 | 多人抢同一剩余名额 |
| 界面测试 | 页面展示、状态颜色、表单校验 |
| 接口测试 | 请求参数、响应结构、异常提示 |

## 3. 核心测试用例

### 3.1 学员预约主教练成功

前置条件：

- 学员 A 的主教练为教练 1；
- 教练 1 明天上午有排班；
- 最大人数 5，当前人数 3。

操作：

- 学员 A 预约明天上午。

期望结果：

- 预约成功；
- actual_coach_id = main_coach_id；
- is_adjusted = 0；
- 排班 current_students + 1。

### 3.2 主教练满员后调剂成功

前置条件：

- 学员 A 主教练教练 1 已满员；
- 教练 2 同时间段还有 3 个名额。

操作：

- 学员 A 选择调剂预约教练 2。

期望结果：

- 预约成功；
- main_coach_id = 教练 1；
- actual_coach_id = 教练 2；
- is_adjusted = 1；
- 教练 2 排班人数 + 1；
- 学员主教练归属不变。

### 3.3 重复预约失败

前置条件：

- 学员 A 已预约明天上午。

操作：

- 学员 A 再次预约明天上午。

期望结果：

- 预约失败；
- 返回“你已预约该时间段，不能重复预约”。

### 3.4 排班满员失败

前置条件：

- 教练 1 最大人数 5，当前人数 5。

操作：

- 学员预约教练 1。

期望结果：

- 预约失败；
- 返回“该时间段已满员”。

### 3.5 取消预约释放名额

前置条件：

- 学员 A 已预约教练 1；
- 排班 current_students = 5。

操作：

- 学员 A 取消预约。

期望结果：

- 预约状态改为 CANCELLED；
- 排班 current_students = 4；
- 可再次被其他学员预约。

### 3.6 完成练车累计次数

前置条件：

- 学员 A 已预约并到场；
- completed_training_count = 3。

操作：

- 教练确认完成练车。

期望结果：

- 预约状态改为 COMPLETED；
- 新增 training_record；
- completed_training_count = 4。

### 3.7 并发抢最后一个名额

前置条件：

- 某排班只剩 1 个名额；
- 两个学员同时预约。

操作：

- 并发提交两个预约请求。

期望结果：

- 只有一个预约成功；
- 另一个返回已满员；
- current_students 不超过 max_students。

## 4. 权限测试

| 测试项 | 期望 |
|---|---|
| 学员访问管理员接口 | 拒绝 |
| 学员取消他人预约 | 拒绝 |
| 教练查看其他教练预约名单 | 拒绝 |
| 管理员查看全部数据 | 允许 |
| 未登录访问业务接口 | 拒绝 |

## 5. 前端验收标准

1. 登录后根据角色进入对应首页；
2. 学员预约页能清晰看到主教练和调剂教练；
3. 满员、可预约、已完成、已取消等状态颜色区分明显；
4. 表单有基础校验；
5. 预约成功和失败均有明确提示；
6. 页面不能出现大量默认模板痕迹；
7. 关键业务组件代码有中文注释。

## 6. 后端验收标准

1. 接口返回统一格式；
2. 关键业务使用事务；
3. 预约并发不会超额；
4. 角色权限控制有效；
5. 核心代码有中文注释；
6. 异常提示为中文；
7. 数据库字段有中文 COMMENT；
8. README 中能说明如何启动项目。

## 7. 答辩验收标准

答辩演示至少包含：

1. 管理员创建教练、车辆和排班；
2. 学员查看主教练预约名额；
3. 学员预约成功；
4. 主教练满员时系统推荐调剂；
5. 教练查看预约名单并完成练车；
6. 管理员查看教练工作量统计。
