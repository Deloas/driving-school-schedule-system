# 项目名称

也可以在页面标题中使用更专业的名称：

# 一、项目背景与边界

本系统用于驾校内部学员练车管理，核心是解决学员练车预约混乱、教练工作量不均衡、车辆资源分配不合理的问题。

传统方式中，学员要练车需要给教练打电话或在微信群接龙，容易出现某一天很多人来、某一天没人来，导致学员体验差、教练工作量不均衡。

本系统必须严格聚焦“练车预约与调度”。

必须做：

1. 学员管理；
2. 教练管理；
3. 车辆管理；
4. 教练排班；
5. 学员预约练车；
6. 预约容量限制；
7. 主教练负责制；
8. 主教练满员后的临时调剂；
9. 教练查看预约名单；
10. 教练确认完成练车；
11. 学员练车次数统计；
12. 管理员统计看板。

严禁偏题：

1. 不做考试预约；
2. 不做考试成绩查询；
3. 不做交管系统对接；
4. 不做复杂缴费合同；
5. 不把系统做成大而全的驾校 ERP；
6. 不加入与练车预约无关的功能。

# 四、核心业务规则

## 1. 学员和教练关系

- 一个教练可以负责多个学员；
- 一个学员只能有一个主教练；
- 学员预约时优先预约主教练；
- 主教练满员时，可以临时调剂到其他教练；
- 调剂不改变学员主教练归属；
- 预约表中必须同时记录 main_coach_id 和 actual_coach_id。

## 2. 预约时间段

系统按半天预约：

- MORNING：上午；
- AFTERNOON：下午。

如果前端做“全天预约”，后端应拆成上午和下午两个时间段处理。

## 3. 教练容量限制

每个教练每个日期每个时间段有最大带练人数。

预约时必须校验：

- 排班是否存在；
- 排班是否开放；
- 当前人数是否小于最大人数；
- 学员是否重复预约；
- 车辆是否正常；
- 是否需要调剂。

## 4. 调剂机制

当学员主教练满员时：

- 查询同日期同时间段其他有空余名额的教练；
- 按剩余名额、当天负载、本周工作量排序；
- 返回推荐调剂教练列表；
- 学员确认后创建调剂预约；
- 保存 is_adjusted = true；
- main_coach_id 保存学员主教练；
- actual_coach_id 保存本次实际带练教练。

## 5. 练车完成

教练可以：

- 查看当天预约名单；
- 确认学员到场；
- 完成练车；
- 填写训练内容；
- 生成 training_record；
- 学员 completed_training_count + 1。

# 五、Redis 使用要求

可以使用 Redis，但 Redis 不是主数据库。

Redis 用于：

1. 预约分布式锁；
2. 可预约排班缓存；
3. 防重复点击限流；
4. 候补队列，可作为加分项；
5. Token 黑名单，可选。

预约接口必须重点实现：

- 使用 Redisson 分布式锁；
- 锁粒度为 scheduleId；
- 锁内重新查询数据库；
- 使用数据库条件更新作为兜底；
- 确保并发预约不会超过 max_students。

核心伪流程：

1. 获取 lock:schedule:{scheduleId}；
2. 查询排班；
3. 判断是否 OPEN；
4. 判断 current_students < max_students；
5. 判断学员是否重复预约；
6. 数据库条件更新 current_students + 1；
7. 插入预约记录；
8. 提交事务；
9. 删除相关缓存；
10. 释放锁。
