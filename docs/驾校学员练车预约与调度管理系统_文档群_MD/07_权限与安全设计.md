# 07 权限与安全设计

## 1. 角色模型

系统包含三类角色：

| 角色 | 说明 |
|---|---|
| ADMIN | 管理员 |
| COACH | 教练 |
| STUDENT | 学员 |

## 2. 权限矩阵

| 功能 | ADMIN | COACH | STUDENT |
|---|---|---|---|
| 学员管理 | 是 | 查看负责学员 | 查看自己 |
| 教练管理 | 是 | 查看自己 | 查看主教练 |
| 车辆管理 | 是 | 查看关联车辆 | 否 |
| 排班管理 | 是 | 查看自己排班 | 查看可预约排班 |
| 创建预约 | 管理员代约 | 否 | 是 |
| 取消预约 | 是 | 否 | 只能取消自己 |
| 调剂预约 | 是 | 查看结果 | 是 |
| 完成练车 | 否 | 是 | 否 |
| 统计看板 | 是 | 查看自己 | 查看个人 |

## 3. 认证方案

推荐使用：

```text
Spring Security + JWT
```

流程：

1. 用户登录；
2. 后端校验账号密码；
3. 登录成功生成 JWT；
4. 前端将 Token 存储到 localStorage 或 sessionStorage；
5. 请求接口时放到 Header；
6. 后端解析 Token 并设置用户上下文。

请求头：

```http
Authorization: Bearer <token>
```

## 4. 密码安全

密码必须加密存储。

推荐使用：

```text
BCryptPasswordEncoder
```

禁止明文存储密码。

## 5. 接口鉴权

示例：

```java
// 管理员才能新增教练
@PreAuthorize("hasRole('ADMIN')")
@PostMapping("/coaches")
public Result<Void> createCoach(@RequestBody CoachCreateDTO dto) {
    return coachService.createCoach(dto);
}
```

## 6. 数据权限

### 6.1 学员数据权限

学员只能查看和操作自己的预约。

校验逻辑：

```text
当前登录用户 related_id == reservation.student_id
```

### 6.2 教练数据权限

教练只能查看：

- 自己负责的学员；
- 自己排班下的预约；
- 调剂到自己名下的预约。

### 6.3 管理员数据权限

管理员可以查看全部数据。

## 7. 防刷与限流

建议使用 Redis 做简单限流：

| 场景 | 限流规则 |
|---|---|
| 登录 | 同一 IP 每分钟最多 10 次 |
| 预约 | 同一学员 5 秒内最多提交 1 次 |
| 查询可预约排班 | 可缓存，不严格限流 |

## 8. 操作日志

建议记录以下操作：

- 管理员新增/修改/删除学员；
- 管理员新增/修改排班；
- 学员创建预约；
- 学员取消预约；
- 管理员手动调剂；
- 教练完成练车。

日志字段：

- 操作人；
- 操作类型；
- 业务类型；
- 操作内容；
- IP；
- 时间。

## 9. 安全注意事项

1. 后端不要完全信任前端传入的角色；
2. 创建预约时，studentId 应从 Token 中获取，不应由前端传；
3. 主教练 ID 应从数据库查询，不应由前端传；
4. 是否调剂应由后端根据 schedule.coach_id 和 student.main_coach_id 判断；
5. 所有修改类接口必须鉴权；
6. 重要业务异常要返回明确中文提示。
