# 12 答辩展示脚本

## 1. 开场说明

各位老师好，我们小组的项目是《基于教练资源均衡的驾校学员练车预约与调度管理系统》。

这个系统不是做考试管理，也不是做交管系统对接，而是聚焦驾校内部日常练车过程，解决传统驾校通过电话、微信群接龙预约练车时存在的预约混乱、人数不均、教练负载不平衡、学员体验差等问题。

## 2. 业务痛点说明

传统方式下，学员如果想练车，需要联系教练或在微信群里接龙。

这会带来几个问题：

1. 某一天来很多学员，每个人练车时间很短；
2. 某一天来的人很少，教练和车辆资源浪费；
3. 教练需要人工统计，效率低；
4. 学员不知道还有没有名额；
5. 主教练满员时，缺少灵活调剂机制。

因此我们系统的目标是：

> 让学员预约更方便，让教练工作量更均衡，让练车资源分配更合理。

## 3. 系统角色介绍

系统分为三个角色：

- 管理员：管理学员、教练、车辆、排班和统计；
- 学员：查看可预约时间，预约练车，接受调剂；
- 教练：查看预约名单，确认练车完成。

## 4. 演示流程

### 4.1 管理员创建基础数据

演示内容：

1. 新增教练；
2. 新增车辆；
3. 新增学员；
4. 为学员绑定主教练；
5. 创建教练排班；
6. 设置每个教练每半天最大带练人数。

讲解重点：

> 系统通过排班容量限制，控制每个教练每个时间段最多带多少学员，避免无限制预约。

### 4.2 学员预约主教练

演示内容：

1. 学员登录；
2. 进入预约页面；
3. 选择明天上午；
4. 系统展示主教练可预约状态；
5. 点击预约成功。

讲解重点：

> 系统优先支持学员预约自己的主教练，符合驾校常见的教练负责制。

### 4.3 主教练满员后的调剂

演示内容：

1. 让主教练排班满员；
2. 学员再次选择该时间段；
3. 系统提示主教练已满；
4. 展示其他空闲教练；
5. 学员选择调剂预约；
6. 预约成功。

讲解重点：

> 调剂不会改变学员的主教练归属，只是本次练车临时由其他教练带练。预约表中同时记录 main_coach_id 和 actual_coach_id。

### 4.4 教练查看预约名单

演示内容：

1. 教练登录；
2. 查看今日预约名单；
3. 识别普通学员和调剂学员；
4. 确认学员到场；
5. 完成练车记录。

讲解重点：

> 教练端不只是查看名单，还可以完成练车确认，系统自动累计学员练车次数。

### 4.5 管理员查看统计

演示内容：

1. 查看今日预约人数；
2. 查看教练工作量对比；
3. 查看调剂预约数量；
4. 查看学员练车次数。

讲解重点：

> 统计数据体现系统的信息化价值，帮助驾校发现教练负载是否均衡。

## 5. 项目技术亮点

1. 前后端分离；
2. Spring Boot + MySQL + Redis；
3. Redis 分布式锁防止超额预约；
4. 数据库条件更新作为并发兜底；
5. 主教练负责制和临时调剂机制；
6. 前端使用产品级设计，不是普通后台模板；
7. 关键业务代码有中文注释。

## 6. 和普通信息管理系统的区别

普通系统主要是增删改查，而我们系统的重点是：

- 预约容量控制；
- 教练负载均衡；
- 学员体验优化；
- 车辆资源校验；
- 调剂推荐；
- 并发防超额。

## 7. 结尾总结

本项目严格围绕老师提出的练车管理需求设计，没有偏向考试管理。系统通过在线预约、教练排班、容量控制、车辆管理和调剂机制，解决了驾校练车预约混乱的问题，体现了信息化系统在真实业务场景中的价值。
