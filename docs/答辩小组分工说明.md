# 答辩小组分工说明

## 分工原则

按业务模块和技术层分工，不按"一人一个页面"简单拆分。每人对自己的模块从后端到前端都能讲解。

## 五人分工

| 成员 | 角色 | 负责模块 |
|------|------|---------|
| 成员 A | 项目负责人 | 项目背景、需求分析、系统边界、整体业务流程、答辩开场与总结 |
| 成员 B | 后端基础 | 登录认证(JWT/SpringSecurity)、角色权限、账号与人员绑定 |
| 成员 C | 排班预约核心 | 教练排班、学员预约、容量控制、Redis分布式锁、并发预约安全 |
| 成员 D | 调剂与练车闭环 | 主教练满员调剂、教练完成练车、标记缺席、练车记录 |
| 成员 E | 前端与部署 | 三端页面、统计驾驶舱、Docker部署、功能验收 |

## 成员 A：项目负责人

**负责讲解：**
- 项目背景（传统电话/微信群接龙问题）
- 需求分析（老师要求聚焦练车不做考试）
- 系统边界（做什么与不做什么）
- 整体业务流程（预约→排班→调剂→完成→记录）
- 答辩开场（1分钟项目介绍）
- 答辩总结（项目亮点回顾）

**准备页面：**
- 登录页（展示品牌设计）
- 管理员驾驶舱首页（展示全局数据）

## 成员 B：后端基础与权限

**负责讲解：**
- JWT 登录认证流程
- Spring Security 配置
- 角色权限体系（ADMIN/COACH/STUDENT）
- 账号与人员绑定（sys_user.related_id → coach/student.id）
- 权限拦截（403 JSON 返回）
- 架构原则：功能模块包含权限分级

**准备页面：**
- 登录页（演示不同角色登录跳转）
- 账号管理页（创建/停用/重置密码）

## 成员 C：排班与预约核心

**负责讲解：**
- 排班管理（日期+时段+容量+批量生成）
- 学员预约流程（查询主教练→预约→容量校验）
- 容量控制（max_students / current_students）
- Redis 分布式锁（lock:schedule:{id}）
- 数据库条件 UPDATE 兜底
- 并发预约安全演示（PowerShell 脚本）

**准备页面：**
- 排班管理页
- 学员预约页（展示主教练卡片+容量进度条）
- 运行并发测试脚本

## 成员 D：调剂与练车闭环

**负责讲解：**
- 主教练满员后的调剂推荐
- 调剂预约（main_coach_id ≠ actual_coach_id）
- 教练完成练车（生成 training_record）
- 标记缺席（不生成记录）
- 学员练车记录（训练档案）
- 学员 completed_training_count 累加

**准备页面：**
- 学员预约页调剂推荐区域
- 教练工作台（完成/缺席操作）
- 学员训练档案页

## 成员 E：前端体验与部署

**负责讲解：**
- 前端技术栈（Vue3/Vite/TypeScript/Tailwind）
- 三端品牌统一设计
- 管理员驾驶舱
- 响应式页面与组件化
- Docker Compose 启动 MySQL/Redis
- IDEA 启动后端
- 功能验收清单

**准备页面：**
- 管理员驾驶舱（统计看板）
- 展示 Docker/IDEA/前端启动流程

## 答辩讲解顺序

### 8-10 分钟版本

| 时间 | 成员 | 内容 |
|------|------|------|
| 0-1min | A | 开场 + 项目背景 |
| 1-2min | B | 登录 + 权限演示 |
| 2-4min | C | 排班 + 预约 + 并发安全 |
| 4-6min | D | 调剂 + 练车闭环 |
| 6-8min | E | 前端设计 + 部署 |
| 8-9min | C | 并发脚本演示 |
| 9-10min | A | 总结 + 亮点回顾 |

### 5 分钟压缩版本

| 时间 | 成员 | 内容 |
|------|------|------|
| 0-1min | A | 开场 |
| 1-2min | C | 预约+并发 |
| 2-3min | D | 调剂+练车 |
| 3-4min | E | 前端+部署 |
| 4-5min | A | 总结 |

## 个人准备建议

每人在答辩前：
1. 在自己电脑上独立启动一遍系统
2. 准备一段 30 秒的模块讲解话术
3. 准备一个可能被老师追问的问题及答案
4. 确认自己负责的页面能正常操作
