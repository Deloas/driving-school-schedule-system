# 项目亮点与答辩话术

## 1 分钟项目介绍

"我们的项目是**驾校学员练车预约与调度管理系统**，核心解决传统电话预约和微信群接龙导致的预约混乱、教练负载不均的问题。

系统围绕**练车预约全流程**：学员在线预约主教练，主教练满员后自动推荐调剂方案，教练确认到场并完成练车，系统自动生成训练记录并累加学员练车次数。整个过程由 Redis 分布式锁保障并发预约不会超额。

系统不做考试管理和交管对接，严格聚焦驾校内部练车调度，这与老师的要求完全一致。"

## 3 分钟项目介绍

"(上接 1 分钟)...技术上采用 Spring Boot + Vue 3 前后端分离架构。

后端使用 Spring Security + JWT 实现角色认证，Redisson 分布式锁防止并发预约超额，数据库条件 UPDATE 作为兜底保障。预约表通过 main_coach_id 和 actual_coach_id 分离设计，既保证主教练负责制，又支持临时调剂。

前端遵循半导(Semi Design)风格，三端(管理员、教练、学员)独立工作台，品牌统一、视觉清爽。管理员有统计驾驶舱；教练有工作台和我的学员；学员有练车工作台和训练档案。

系统核心亮点包括：主教练负责制和调剂机制、Redis 双层防超额、预约到训练记录的完整闭环，以及产品级的前端体验。"

## 为什么不做考试？

"老师录音中明确要求本系统重点管理驾校内部**学生练车过程**，不管理考试。

考试预约、成绩查询属于交管考试系统范畴。如果引入考试模块，系统范围会过度扩大，偏离核心，也会让答辩重点分散。

因此我们**严格聚焦练车预约与调度**，让系统边界清晰、业务深度到位。"

## 如何解决微信群接龙预约问题？

"传统方式下学员要练车时给教练打电话或在微信群接龙，容易出现某天大量学员集中到场，每人练车时间不足，而另一天无人到场，教练和车辆资源闲置。

我们通过排班容量控制解决：每个教练每半天有明确的 max_students 上限。学员按日期和时段在线预约，系统自动校验容量，满员后提示并推荐其他可用教练。

这样既保障学员练车体验，也让教练每日工作量更均衡。"

## 如何解决预约超额？

"我们采用双重保障：

1. **Redisson 分布式锁**：以 scheduleId 为粒度加锁，锁内重新查数据库并校验所有条件；
2. **数据库条件 UPDATE**：`UPDATE coach_schedule SET current_students=current_students+1 WHERE id=? AND OPEN AND current<max`，如果影响行数为 0 则回滚。

这样即使 Redis 锁失效或网络延迟，数据库条件 UPDATE 也能兜底防止超额。"

## 如何体现权限分级？

"我们采用**功能模块包含权限分级**的设计方式。

每个功能模块(如 coach、student、reservation)是统一的，不是按角色拆成三套重复代码。模块内部通过 `@PreAuthorize` 和 `SecurityContextUtils` 实现权限控制：

- 管理员有写权限
- 教练只能查看和操作自己的数据
- 学员只能操作自己的预约和记录

前端也通过路由守卫 + 请求拦截器实现同样的权限隔离。"

## 如何体现工程质量？

1. **分层架构**：Controller → Service → Mapper，DTO/VO 分离
2. **统一异常处理**：GlobalExceptionHandler 统一捕获并返回中文提示
3. **统一响应体**：所有接口返回 `{ code, message, data }`
4. **事务管理**：创建/取消预约使用 `@Transactional`
5. **幂等初始化**：DataInitializer 支持多次重启不重复插入
6. **200+ 中文注释**：所有核心代码包含业务语义注释
7. **Git 规范**：10 次 commit，按阶段提交，message 清晰

## 老师可能追问的问题

### Q: Redis 锁为什么用 Redisson？

A: Redisson 基于 Netty，支持自动续期和 watch dog 机制。相比手写 SETNX，Redisson 更可靠，不会因为业务执行时间长导致锁提前释放。

### Q: 条件 UPDATE 为什么还要保留？

A: 分布式锁只能保证同一 JVM 的并发安全。如果将来部署多实例，数据库条件 UPDATE 仍能兜底。这是双重保险的设计思路。

### Q: main_coach_id 和 actual_coach_id 为什么分开？

A: 学员归属主教练不变，但某次练车可能由其他教练临时带。这个字段分离体现了主教练负责制和调剂机制的共同存在。调剂预约的 is_adjusted=1 标记也可以用于管理员查看调剂率。

### Q: 为什么不用 WebSocket 做实时通知？

A: 本系统面向驾校内部管理，实时性要求不高。页面刷新和 Toast 提示已能满足需求。WebSocket 可作为后续扩展。

### Q: 表里为什么有 operation_log？

A: 预留操作日志能力，虽然当前没有全量记录，但数据库已经具备审计追溯的基础。
