# 答辩常见问题与回答

## 1. 为什么不做考试功能？

老师录音中明确要求系统重点管理驾校内部学生练车过程，不管理考试。考试预约和成绩属于交管考试系统范围。如果引入考试模块，系统范围会过度扩大，偏离核心问题——练车预约与调度。我们选择把"练车安排"这个高频痛点做深，而不是泛泛做一个大而全的系统。

## 2. 为什么选择练车预约作为核心？

练车是驾校日常运营中频率最高、最容易产生混乱的环节。学员、教练、车辆、时间四者之间需要协调。传统电话和微信群方式效率低且容易出错。信息化预约能解决这个核心矛盾，而且老师明确要求我们聚焦这个方向。

## 3. 如何防止预约超额？

采用双重保障。第一层：Redisson 分布式锁，以 scheduleId 为粒度，加锁后重新查数据库校验容量和重复预约。第二层：数据库条件 UPDATE 语句 `UPDATE coach_schedule SET current_students = current_students + 1 WHERE id = ? AND status = 'OPEN' AND current_students < max_students`。条件 UPDATE 是原子操作，影响行数为 0 时会回滚整个事务，保证数据库层面也不会超额。

## 4. Redis 锁失效怎么办？

数据库条件 UPDATE 是最终兜底。即使 Redis 不可用或锁机制出现问题，UPDATE 语句的 WHERE 条件中的 `current_students < max_students` 会阻止超额更新。MySQL 的行级锁保证了这条 UPDATE 的原子性。

## 5. 为什么还要数据库条件 UPDATE？

分布式锁只能保证同一时刻只有一个 JVM 线程执行。但如果将来部署多个实例，或者 Redis 出现网络延迟，条件 UPDATE 作为数据库层面的兜底保障更加可靠。这是防御性设计。

## 6. 主教练和实际教练有什么区别？

main_coach_id 是学员的归属教练，固定不变。actual_coach_id 是本次练车实际带练的教练。正常情况下两者相同；当主教练满员、学员选择调剂后，actual_coach_id 会指向调剂教练，main_coach_id 保持不变。这个设计同时支撑了主教练负责制和灵活调剂。

## 7. 调剂预约如何实现？

当学员查询预约方案时，系统检测主教练排班满员。然后查询同日期、同时段、OPEN 状态、有空位的其他教练排班。按剩余名额从多到少排序展示。学员确认后，创建一条 is_adjusted=1 的预约记录，main_coach_id 保留原主教练，actual_coach_id 存调剂教练。调剂预约和普通预约走同一套 Redis 锁和事务保护。

## 8. 为什么不让学生自由选择任意教练？

自由选择会导致教练工作量更不均衡——热门教练被抢满，冷门教练无人问津。我们设计的主教练负责制让学员有固定的归属教练，同时通过调剂机制在主教练满员时提供弹性。这种方式既保证了教练负责制，又有灵活性。

## 9. 教练如何查看未来预约？

教练工作台提供日期筛选功能。默认显示今天，可以点击"明天"或选择任意日期查看对应日期的预约名单。接口支持 date 参数传入任意日期。

## 10. 学员缺席如何处理？

教练在预约名单中点击"缺席"按钮，填写缺席原因。系统将预约状态改为 ABSENT。缺席不生成训练记录，不增加学员的练车次数。coach_schedule 的 current_students 也不会减少——名额已经被占用。

## 11. 练车记录如何生成？

教练点击"完成练车"后，系统在一个事务中完成三步：更新预约状态为 COMPLETED；插入一条 training_record 记录训练内容、时长、教练评语；将对应学员的 completed_training_count 加 1。三步要么全部成功，要么全部回滚。

## 12. 权限是如何控制的？

我们采用"功能模块包含权限分级"的设计方式。每个模块（如 coach、student、reservation）是统一的，不按角色拆分。Controller 层通过 @PreAuthorize 注解实现方法级权限控制，Service 层通过 SecurityContextUtils 从 Token 中获取当前用户身份和绑定关系，动态过滤数据。

## 13. 为什么采用 JWT？

JWT 适合前后端分离的无状态架构。Token 本身包含了用户 ID、角色、绑定关系 ID 等信息，服务端不需要存储会话。JWT 的过期机制天然支持登录超时。同时前端可以在 localStorage 中持久化 Token，页面刷新后不影响登录状态。

## 14. 账号和教练/学员如何绑定？

sys_user 表的 related_id 字段指向对应的 coach.id 或 student.id。例如 coach002 的 related_id=2 指向张三的教练记录。登录时 Token 中会包含 relatedId，后续接口通过这个值确定当前用户能访问哪些数据。

## 15. 数据库有哪些核心表？

8 张表。sys_user 存登录账号，coach/student/vehicle 存基础资料，coach_schedule 存排班和容量，reservation 存预约记录（核心表，支持调剂），training_record 存练车记录，operation_log 存操作日志。

## 16. 项目如何体现软件工程思想？

分层架构（Controller→Service→Mapper）、统一异常处理（GlobalExceptionHandler）、统一响应体（Result）、事务管理（@Transactional）、幂等初始化（DataInitializer）、代码注释（所有核心代码含中文注释）、Git 分支管理（按阶段提交）。

## 17. 前端做了哪些产品化优化？

品牌统一（Logo、色彩、圆角、阴影）、三端工作台（管理员驾驶舱、教练工作台、学员练车工作台）、统一组件（StatusBadge、EmptyState、PageModal、PageHeader）、筛选区卡片化、表格记录计数、友好的空状态提示、密码重置、账号停用二次确认。

## 18. 如果多人同时预约怎么办？

通过 Redisson 分布式锁保证同一排班在同一时间只有一个线程能执行预约逻辑。锁内重新查数据库校验所有条件。未被获取到锁的线程会返回"当前预约人数较多，请稍后重试"。同时 5 秒限流防止同一学员重复点击。

## 19. 如何证明系统没有加入考试功能？

整个项目代码中不包含任何考试预约、考试成绩、交管系统对接的接口、页面、数据库字段。系统边界在 README 和所有文档中明确声明。数据库 8 张表全部围绕练车管理设计。

## 20. 后续还可以怎么扩展？

消息通知（预约成功通知、教练提醒）、数据报表导出（Excel 导出练车统计）、更细粒度的智能调度推荐算法、移动端适配、部署到云服务器。

## 21. 为什么 coach_schedule 用 current_students 计数，而不是从 reservation 表 COUNT？

直接计数避免了每次查询都要做表关联的 COUNT 操作，性能更好。同时条件 UPDATE `WHERE current_students < max_students` 利用数据库行级锁的原子性来防止超额。这种方式在并发场景下比 SELECT COUNT 更可靠。

## 22. 满员后 current_students 会不会超过 max_students？

不会。有两层保障：Redis 锁确保同一时间只有一个线程操作同一排班；数据库条件 UPDATE 的 WHERE 条件 `current_students < max_students` 确保即使锁失效，UPDATE 影响行数为 0 时会触发异常并回滚事务。

## 23. 项目前后端如何通信？

通过 RESTful API，统一返回 `{ code, message, data }` 格式。前端 Axios 拦截器自动附加 JWT Token，响应拦截器检测 code !== 200 时自动抛出异常，触发 Toast 错误提示。

## 24. 系统如何启动？

Docker Compose 一键启动 MySQL 和 Redis。IDEA 运行 Spring Boot 主类启动后端。前端 npm run dev 启动 Vite 开发服务器。三个命令即可启动整个系统。

## 25. 小组如何分工？

五人分别负责：登录认证与账号权限、基础资料管理、排班管理、学员预约与调剂、练车闭环与统计看板。按业务功能模块分工，每人对自己的模块从数据库到前端都能讲解。
