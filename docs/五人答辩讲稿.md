## 成员 A：项目负责人

### 30 秒讲稿

"我们的项目是驾校学员练车预约与调度管理系统。核心解决传统驾校用电话和微信群预约练车带来的预约混乱、教练负载不均的问题。系统聚焦驾校内部练车管理，严格遵循老师要求，不做考试管理。接下来由我们小组各成员介绍系统的不同模块。"

### 1 分钟讲稿

"大家好，我们小组的项目是驾校学员练车预约与调度管理系统。

在传统驾校中，学员要练车需要给教练打电话或在微信群接龙。这种方式容易出现某天大量学员集中到场、每人练车时间不足、另一天无人到场导致教练和车辆闲置的问题。

我们设计的系统围绕练车预约全流程：学员在线预约主教练，系统通过容量控制防止超额；主教练满员后自动推荐其他可用教练进行临时调剂；教练确认完成练车后，系统自动生成训练记录并累加学员练车次数。

系统严格聚焦练车管理，不做考试预约、不做成绩管理、不做交管系统对接，这正是老师对我们的核心要求。"

### 可能被问的问题

**Q: 为什么不把考试管理也做了？**
A: 老师录音中明确要求系统重点管理驾校内部学生练车过程。考试预约、成绩属于交管考试系统范围。如果加入考试，系统边界会过度扩大，也偏离了我们解决的核心问题——练车预约调度。

## 成员 B：后端基础与权限

### 30 秒讲稿

"我负责系统的登录认证和权限控制。系统三类角色——管理员、教练、学员——通过 JWT Token 实现无状态认证。权限采用功能模块内部区分角色的设计，避免了按角色重复开发三套独立系统。同时实现了账号与教练、学员的一对一绑定，停用账号后无法登录。"

### 1 分钟讲稿

"我负责系统的登录认证和权限控制。

系统使用 Spring Security + JWT 实现无状态认证。用户登录后获得 Token，后续所有请求通过拦截器自动附加，后端从 Token 中解析用户身份和角色。

权限设计上，我们采用'功能模块包含权限分级'的方式。每个业务模块如教练管理、学员管理是统一的，不是按角色拆成三套。模块内部通过 @PreAuthorize 注解实现方法级权限控制，管理员有写权限，教练和学员只能查看和操作自己的数据。

账号管理模块实现了教练/学员账号的一对一绑定——sys_user 表的 related_id 字段指向对应的 coach.id 或 student.id。停用账号后，登录接口会返回'该账号已被禁用'。

前端也通过路由守卫和请求拦截器实现了同样的权限隔离。"

### 可能被问的问题

**Q: 为什么用 JWT 而不是 Session？**
A: JWT 无状态，服务端不需要存储会话信息。Token 本身包含用户 ID、角色、绑定关系等，解析即可获得用户身份，适合前后端分离架构。同时 JWT 过期机制天然支持登录超时。

## 成员 C：排班预约与并发安全

### 30 秒讲稿

"我负责排班管理和学员预约模块。管理员可以按日期、时段、教练和车辆创建排班，支持批量生成。学员选择日期和时间段后，系统优先查询主教练排班，展示剩余名额。预约时通过 Redisson 分布式锁和数据库条件 UPDATE 双重保障，确保并发场景下不会超额预约。"

### 1 分钟讲稿

"我负责排班管理和学员预约模块。

排班是预约的前置条件。管理员可以为每个教练按日期和时段创建排班，设置最大带教人数，并绑定训练车辆。系统支持批量生成——选中多个教练、日期范围和时段，一键创建。

学员预约时，选择日期和时段后点击查询，系统展示主教练的排班信息：教练姓名、车辆、当前预约人数、剩余名额和容量进度条。点击立即预约，系统进行多重校验。

最核心的是并发安全设计。假设排班还剩最后一个名额，两个学员同时点击预约，我们通过两层保障防止超额：第一层，Redisson 分布式锁以 scheduleId 为粒度，锁内重新查数据库校验所有条件；第二层，数据库条件 UPDATE 语句 `WHERE current_students < max_students` 作为兜底。即使 Redis 锁出现问题，数据库层面也不会超额。

同时我们还加入了 5 秒的学员级限流，防止重复点击。"

### 可能被问的问题

**Q: Redis 锁失效怎么办？**
A: 数据库条件 UPDATE 是最终兜底。UPDATE 语句本身是原子操作，WHERE 条件中的 `current_students < max_students` 保证了即使多个事务同时执行，也只有一个能成功修改。更新影响行数为 0 时，代码会抛出异常并回滚事务。

## 成员 D：调剂机制与练车闭环

### 30 秒讲稿

"我负责主教练满员后的调剂机制和练车完成闭环。当学员主教练满员时，系统自动推荐同时间段有空位的其他教练。学员确认后创建调剂预约——系统保留主教练归属，同时记录实际带练教练。教练完成练车后，系统生成训练记录并累加学员练车次数，形成完整闭环。"

### 1 分钟讲稿

"我负责调剂机制和练车闭环两个模块。

第一个是调剂机制。每个学员都归属一个主教练，预约时优先预约主教练。但当主教练满员时，学员的练车需求仍然需要满足。我们设计了调剂推荐功能：系统查询同日期、同时段、状态正常且有空位的其他教练，按剩余名额从多到少排序展示。学员可以选择其中一个教练进行临时调剂。

数据库层面，预约表的 main_coach_id 始终记录学员的主教练，actual_coach_id 记录本次实际带练的教练。is_adjusted 字段标记是否为调剂预约。这样既保证了主教练负责制，又支持了灵活的临时调度。

第二个是练车闭环。学员预约到场后，教练在管理端可以看到今日预约名单。点击'完成练车'，填写练车内容和评语，系统会：更新预约状态为已完成；生成一条 training_record 记录训练详情；同时学员的已完成练车次数加 1。如果学员未到场，教练可以标记缺席——不生成训练记录，不累加次数。"

### 可能被问的问题

**Q: 调剂后学员的主教练会改变吗？**
A: 不会。main_coach_id 始终指向学员的原主教练，调剂只是本次练车由其他教练临时带。学员的归属关系不会因为调剂而改变。

## 成员 E：前端体验、统计看板与部署

### 30 秒讲稿

"我负责前端产品体验优化和统计驾驶舱。三端——管理员、教练、学员——都有独立的工作台页面，品牌风格统一。管理员有完整的统计驾驶舱，可以看到教练负载、预约趋势、学员进度等数据。系统通过 Docker Compose 一键启动 MySQL 和 Redis，整体部署简洁。"

### 1 分钟讲稿

"我负责前端产品体验和统计驾驶舱。

我们的前端采用 Vue 3 + TypeScript + Vite + Tailwind CSS 技术栈。为了让系统看起来更像正式产品而不是课程作业，我们在视觉上做了统一的品牌设计：方向盘造型的 Logo、角色头像标识、统一的筛选卡片、表格记录数统计、以及友好的空状态提示。

三端——管理员、教练、学员——都有独立的工作台。管理员的首页是一个统计驾驶舱，展示教练负载排行、近 7 天预约趋势、排班容量利用率、学员练车进度分布和最近动态时间线。教练有个人工作台，可以看到今天的预约、今日完成数、我的学员和最近练车记录。学员有练车工作台，可以看到练车进度统计、下一次预约安排和最近训练记录。

系统的部署也非常简洁。MySQL 和 Redis 通过一个 docker-compose.yml 文件一键启动。后端在 IDEA 中运行，前端 npm run dev，三分钟内可以启动整个系统。"

### 可能被问的问题

**Q: 前端为什么没有用组件库？**
A: 我们希望系统有自己的视觉风格，而不是和千篇一律的模板后台撞脸。Tailwind CSS 提供了足够的原子化样式能力，我们自定义了配色、圆角、阴影等设计 Token，最终效果更接近产品级中后台。
