## 成员 A：登录认证与账号权限模块

### 30 秒讲稿

"我负责系统的登录认证和账号权限管理。系统使用 JWT + Spring Security 实现三类角色的无状态认证。权限采用功能模块内部区分角色的设计，避免了按角色重复开发三套系统。同时实现了账号与教练、学员的一对一绑定，停用账号后无法登录，未登录和权限不足都有规范的 JSON 错误返回。"

### 1 分钟讲稿

"我负责系统的登录认证与账号权限模块。

系统使用 Spring Security + JWT 实现无状态认证。用户登录后获得 Token，后续所有请求通过拦截器自动附加，后端从 Token 中解析用户身份和角色。密码使用 BCrypt 加密存储。

权限设计上，我们采用'功能模块包含权限分级'的方式——每个业务模块如教练管理、学员管理是统一的，不按角色拆成三套。模块内部通过 @PreAuthorize 实现方法级权限控制。关键发现：Spring Security 的 hasRole 方法会自动拼接 ROLE_ 前缀，所以我们在 SecurityContext 中设置权限时也加了 ROLE_ 前缀。

账号管理模块实现了教练/学员账号的一对一绑定——sys_user 表的 related_id 字段指向对应的 coach.id 或 student.id。每个教练或学员只能拥有一个账号。停用账号后，登录接口返回'该账号已被禁用'。"

### 可能被问的问题

**Q: 为什么用 JWT 而不是 Session？**
A: JWT 无状态，Token 本身包含用户 ID、角色、绑定关系等信息，服务端不需要存储会话，适合前后端分离。过期机制天然支持登录超时。

**Q: 账号和人员是什么关系？**
A: sys_user.related_id 指向 coach.id 或 student.id，一对一绑定。创建教练账号时必须选择对应教练，创建后可重置密码或停用。

## 成员 B：基础资料管理模块

### 30 秒讲稿

"我负责系统的基础资料管理模块。教练管理支持新增、编辑、停用和每半天最大带教人数设置。学员管理重点维护学员与主教练的绑定关系，并追踪练车进度。车辆管理记录训练车辆的车牌、状态和教练绑定。三块基础资料为后续的排班和预约提供了核心数据支撑。"

### 1 分钟讲稿

"我负责系统的基础资料管理模块，包括教练管理、学员管理和车辆管理。

教练管理维护教练的姓名、手机号、教练证号，以及默认每半天最大带教人数——这个值会作为排班容量的参考。教练状态支持正常、请假和停用。

学员管理的核心是主教练绑定——每个学员必须归属一个教练，系统通过 main_coach_id 建立这个关系。学员记录还追踪练车进度：应完成次数和已完成次数。超过应完成次数的学员不宜再占用排班资源。

车辆管理记录车牌号、绑定教练、车型和状态。维修中和停用的车辆不能被排班绑定。所有管理页面都支持关键词搜索、状态筛选、新增编辑弹窗，提供统一的 CRUD 操作体验。"

### 可能被问的问题

**Q: 学员和主教练的绑定关系是否可以修改？**
A: 可以。编辑学员时可以更换主教练。系统会校验新教练的状态必须是正常，不能绑定到已停用的教练。

## 成员 C：排班管理模块

### 30 秒讲稿

"我负责排班管理模块。管理员可以按日期、时段、教练和车辆创建排班，设置每个时段的最大带教人数。系统支持批量生成——选中多个教练、多个日期、多个时段，一键生成多条排班。排班有 OPEN、CLOSED、CANCELLED 三种状态，被关闭或取消的排班学员无法预约。"

### 1 分钟讲稿

"我负责排班管理模块。

排班是系统资源组织的核心——它定义了每个教练在特定日期和时间段能带多少学员、用哪辆车。管理员可以单条新增排班，也可以通过批量生成功能，选择多个教练、日期范围和时段，系统自动为每个组合生成一条排班记录。

排班有三个关键校验：一是同一教练同一天同一时间段不能重复创建；二是同一车辆不能在同一时段被多个教练占用；三是编辑排班容量时不能把 max_students 改到小于已预约的 current_students。

排班的 current_students 字段会随预约和取消动态变化——预约成功 +1，取消预约 -1。OPEN 状态的排班才会展示给学员预约。CLOSED 只是关闭新增预约，不影响已有预约；CANCELLED 则彻底取消。"

### 可能被问的问题

**Q: 批量生成排班的逻辑是什么？**
A: 三重循环：coachIds × 日期范围 × timeSlots。对每个组合检查是否已有排班，有则跳过，无则创建。最后返回成功创建和跳过的数量。

## 成员 D：学员预约与调剂模块

### 30 秒讲稿

"我负责主教练满员后的调剂机制和练车完成闭环。当学员主教练满员时，系统自动推荐同时间段有空位的其他教练。学员确认后创建调剂预约——系统保留主教练归属，同时记录实际带练教练。教练完成练车后，系统生成训练记录并累加学员练车次数，形成完整闭环。"

### 1 分钟讲稿

"我负责调剂机制和练车闭环两个模块。

第一个是调剂机制。每个学员都归属一个主教练，预约时优先预约主教练。但当主教练满员时，学员的练车需求仍然需要满足。我们设计了调剂推荐功能：系统查询同日期、同时段、状态正常且有空位的其他教练，按剩余名额从多到少排序展示。学员可以选择其中一个教练进行临时调剂。

数据库层面，预约表的 main_coach_id 始终记录学员的主教练，actual_coach_id 记录本次实际带练的教练。is_adjusted 字段标记是否为调剂预约。这样既保证了主教练负责制，又支持了灵活的临时调度。

第二个是练车闭环。学员预约到场后，教练在管理端可以看到今日预约名单。点击'完成练车'，填写练车内容和评语，系统会：更新预约状态为已完成；生成一条 training_record 记录训练详情；同时学员的已完成练车次数加 1。如果学员未到场，教练可以标记缺席——不生成训练记录，不累加次数。"

### 可能被问的问题

**Q: 调剂后学员的主教练会改变吗？**
A: 不会。main_coach_id 始终指向学员的原主教练，调剂只是本次练车由其他教练临时带。学员的归属关系不会因为调剂而改变。

## 成员 E：练车闭环与统计看板模块

### 30 秒讲稿

"我负责练车完成闭环和统计驾驶舱。教练在预约名单中点击完成练车，系统生成训练记录并累加学员次数；点击缺席则不生成记录。学员有训练档案页可以查看历史练车内容和教练评语。管理员有统计驾驶舱，展示教练负载、预约趋势和学员进度分布。"

### 1 分钟讲稿

"我负责系统的练车闭环和统计看板。

练车闭环是预约流程的终点。教练在工作台看到今天的预约名单，对到场的学员点击完成练车——在一个数据库事务中：预约状态变为已完成，生成一条 training_record 记录训练内容、时长和教练评语，同时对应学员的已完成练车次数加一。如果学员没到场，教练点击标记缺席——状态变为缺席，不生成记录，不增加次数。

学员端有训练档案页，以卡片形式展示每次练车的日期、时段、教练、车辆、训练内容和教练评语。这个页面相当于学员的练车成长记录。

管理员端有统计驾驶舱，包括教练负载排行（利用率进度条）、近七天预约趋势、排班容量利用率、学员练车进度分布和最近动态时间线。教练和学员也有各自的个人看板。

在部署方面，项目通过一个 docker-compose.yml 文件一键启动 MySQL 和 Redis，IDEA 启动后端，npm run dev 启动前端，三分钟即可启动整个系统。"
