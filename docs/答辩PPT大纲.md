# 答辩 PPT 大纲

共 14 页，适合 8-10 分钟答辩讲解。

---

## 第 1 页：封面

**标题：** 驾校学员练车预约与调度管理系统

**副标题：** 面向驾校内部练车资源协调的智能预约与调度平台

**信息：**
- 小组成员：5 人
- 技术栈：Spring Boot + Vue 3 + MySQL + Redis

---

## 第 2 页：项目背景与问题

**传统驾校练车预约的痛点：**

| 问题 | 表现 |
|------|------|
| 微信群接龙混乱 | 学员不知道什么时间有人练车 |
| 电话预约效率低 | 教练需要逐个确认学员意向 |
| 教练工作量不均 | 某天很多人来，某天没人来 |
| 缺少容量控制 | 学员集中到场，每人练车时间不足 |
| 车辆调度随意 | 没有统一的排班和车辆占用管理 |
| 练车无记录 | 练完就完，无法追溯训练历程 |

**老师明确要求：聚焦练车管理，不做考试。**

---

## 第 3 页：项目定位与边界

| 系统做 | 系统不做 |
|--------|---------|
| 教练 / 学员 / 车辆管理 | 考试预约 |
| 排班容量控制 | 考试成绩管理 |
| 学员在线预约练车 | 交管系统对接 |
| 主教练满员调剂 | 缴费与合同 |
| 并发预约安全 | 驾考全流程 |
| 练车完成与记录 | |
| 统计驾驶舱 | |

> 把"练车安排"这个高频痛点做深，而不是泛泛做大而全的系统。

---

## 第 4 页：系统角色与权限

| 角色 | 核心能力 |
|------|---------|
| 管理员 | 全局管理、排班、账号、统计 |
| 教练 | 查看预约、完成练车、我的学员 |
| 学员 | 预约练车、取消、查看训练记录 |

> 架构原则：功能模块包含权限分级，不按角色拆分重复模块。

---

## 第 5 页：系统业务流程

```
账号开通 → 基础资料维护 → 教练排班
  → 学员预约（优先主教练）
  → 容量控制（Redis锁 + 条件UPDATE）
  → 主教练满员 → 调剂推荐 → 学员确认
  → 教练确认到场 → 完成练车 / 标记缺席
  → 生成训练记录 → 累计练车次数
  → 统计看板
```

---

## 第 6 页：核心功能一——基础资料与账号

- **教练管理**：姓名、手机、教练证号、每半天最大带教人数
- **学员管理**：姓名、主教练、练车科目、练车进度
- **车辆管理**：车牌、绑定教练、车型、维修/停用状态
- **账号管理**：为教练和学员开通登录账号，一对一绑定

---

## 第 7 页：核心功能二——排班与预约

**排班：**
- 按日期 + 时段（上午/下午）+ 教练 + 车辆创建
- 支持批量生成（多教练 × 多天 × 多时段）
- 容量控制：每个排班设置 max_students

**预约：**
- 学员选择日期和时段，系统查询主教练排班
- 展示容量进度条（当前人数 / 最大人数）
- 校验重复预约、满员拦截
- 预约成功 → 排班 current_students + 1

---

## 第 8 页：核心功能三——并发安全

**双重保障防止超额：**

| 层级 | 方案 |
|------|------|
| 应用层 | Redisson 分布式锁（粒度 scheduleId） |
| 数据库层 | 条件 UPDATE（`WHERE current < max`） |

**锁内流程：**
```
加锁 → 重新查排班 → 校验容量 → 校验重复
  → 条件 UPDATE → INSERT reservation → 释放锁
```

**防重复点击：** 5 秒限流（同一学员重复提交拦截）

---

## 第 9 页：核心功能四——满员调剂

**场景：** 主教练满员，其他教练有空位

**流程：**
1. 系统检测主教练排班满员
2. 查询同日期+同时间段+OPEN+有空位的其他教练
3. 按剩余名额排序，返回调剂推荐列表
4. 学员选择调剂 → 创建调剂预约

**数据库设计：**
- `main_coach_id` = 学员主教练（不变）
- `actual_coach_id` = 本次实际带练教练
- `is_adjusted` = 1

---

## 第 10 页：核心功能五——练车闭环

| 操作 | 效果 |
|------|------|
| 教练点「完成」 | 状态 → COMPLETED，生成 training_record，学员次数 +1 |
| 教练点「缺席」 | 状态 → ABSENT，不生成记录，不增加次数 |

**学员训练档案：**
- 每次练车的内容、时长、教练评语自动沉淀
- 学员随时查看历史训练记录

---

## 第 11 页：统计驾驶舱

**管理员驾驶舱：**
- 学员/教练/车辆总数
- 今日预约/完成/缺席/剩余名额
- 教练负载排行（利用率进度条）
- 近 7 天预约趋势
- 排班容量利用率
- 学员练车进度分布
- 最近动态时间线

## 第 13 页：数据库设计

**8 张核心表：**

| 表 | 作用 |
|------|------|
| sys_user | 系统登录账号 |
| coach | 教练资料 |
| student | 学员资料 + 练车进度 |
| vehicle | 训练车辆 |
| coach_schedule | 教练排班 + 容量 |
| reservation | 预约记录（核心） |
| training_record | 练车记录 |
| operation_log | 操作日志 |

**关键设计：**
- `sys_user.related_id` → 人员绑定
- `reservation.main_coach_id ≠ actual_coach_id` → 调剂
- `reservation → training_record` → 练车闭环

---

## 第 14 页：总结

**项目价值：**
- 替代微信群和电话预约，让练车安排有序高效
- 容量控制 + 并发锁，确保预约安全
- 满员调剂，资源弹性利用
- 练车完成 → 训练记录，沉淀学员训练过程
- 统计驾驶舱，支撑运营决策

**后续方向：**
- 消息通知（预约成功/教练提醒）
- 数据报表导出
- 更细粒度的智能调度算法
- 部署到服务器环境
